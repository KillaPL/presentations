Introduction to object oriented programming in ruby

Łukasz Strzebińczyk

Ruby is object oriented programming language
Object oriented programming languages are all about objects.
In Ruby
Everything is an object™
Lets look at this piece of code: 

object.method(param)
=> return_value
'object' is an object
'param' is an object
'return_value' is an object
'method' is a method, a mean by which all of these interact with each another.
Analising and crafting data and logic flow through objects and methods is the heart of object-oriented programming.
Objects come in all shapes and sizes.
Different objects are used for different jobs.
but ultimately...
...they are all just an instance of the same, basic thing
In Ruby there is a couple of basic classes of objects:
nil
boolean
number
string
symbol
array
hash
Those are the most basic building blocks of the ruby code.

nil

  • Represents lack of value
  • is falsy
  • ...but a regular object
  • everything responds to #nil? in ruby
  • often returned from methods if their basic feature fails: 
    • 
[1, 2, 3].find{|n| n == 1 }
=> 1
                  
    • 
[1, 2, 3].find{|n| n == 4 }
 => nil
                  
    • 
User.find_by(id: 1)
=> #<User id: 1> 
                  
    • 
User.find_by(id: 100)
=> nil

bool values

  • AKA true/false
  • represent logical truthiness and falsiness
  • always (well, usually) returned from '?' methods
  • 
"string".empty? => false
                
  • 
"".empty? => true

numbers

  • There is a couple of classes describing numbers in ruby: 
    • 
0.class => Fixnum
                  
    • 
0.0.class => Float
                  
    • 
(100**10).class => Bignum
  • basically, they represent numerical value as we know and love

string

  • Strings are sequences of characters, used to represent text data
  • You can easily recognise them: "string", 'another string'
  • Most widely used objects in web applications. Users names, posts contents ands so on, are all strings.
  • They are used to represent the actual text value contained in the characters.

symbol

  • Symbols are the hardest to explain
  • They are not just a fancy strings!
  • You can recognise them by a colon in front of them: :symbol, :another_symbol
  • They are used to represent an idea, concept, which surfaces in the application
  • In rails app you are likely to find :id, :name, :user, :action, :controller and so on.
  • We don't care about their string values, we care about concepts those point us to

arrays

  • Arrays represent a collection of objects
  • They are largely what differs ruby from lower level languages like c, c++ or java
  • They are containers, they contain objects of another classes. You can easily have
    • Arrays of numbers: 
      [1, 2, 3]
    • Arrays of strings: 
      ["a", "b", "c"]
    • Arrays of whatever you like: 
      [1, "a", [""], {s: [1]}]
    • However it's best for your arrays to store objects of just one class!

arrays, part 2

  • Since arrays are containers, they offer a great deal of features
  • Finding elements: 
    
[1, 2, 3].find{|n| n.even? } => 2
  • Sorting : 
    
[2, 6, 4, 8].sort
 => [2, 4, 6, 8]
  • Filtering : 
    
[2, 6, 4, 8].delete_if{|elem| elem > 5 }
 => [2, 4]
  • You can apply some code to each element and return new array: 
    
[1, 2, 3, 4, 5].map{|elem| elem ** 2}
 => [1, 4, 9, 16, 25]
  • And much much more!

hashes

  • Hashes are also containers, but they associate key objects with value objects
  • 
2.2.0 :001 > hash = {key: 'value'}
 => {:key=>"value"}
2.2.0 :002 > hash[:key]
 => "value"
  • Symbols are usually used as keys
  • Basically used as container for data

hashes, part 2

Up to ruby 1.9 hashes were written like this: 
{:key => "value"}
But since ruby 1.9.0 you can write this as 
{key: "value"}
And that's exactly the same thing 

2.2.0 :001 > {:key => 'value'} == {key: 'value'}
 => true
But we rarely use just the primitives. For the most time we have a set of informations, which need to be handled together.
Informations, which we need to...
compose
and...
encapsulate
This is where classes show up.
Each object is an instance of some class: 

2.2.0 :001 > nil.class
=> NilClass 
2.2.0 :002 > "string".class
=> String 
2.2.0 :003 > :symbol.class
=> Symbol
Classes are objects too! 

2.2.0 :006 > String.class
 => Class
We use classes to encapsulate data and behavior.
Say, if we want to represent a person with name and age in code, we can write such class: 

class Person
  def initialize(name, age)
    @name = name
    @age  = age
  end
end
And now we can initialise those: 

2.2.0 :001 > Person.new('Łukasz', 26)
=> #<Person:0x00000001d36160 @name="Łukasz", @age=26>
This is what we call encapsulation of data.
We locked a couple of informations in single container, and we use it as a whole.
If we want to also encapsulate behavior, we add methods.
For example, is the person an adult?

class Person
  def initialize(name, age)
    @name = name
    @age  = age
  end

  def adult?
    @age >= 18
  end
end

2.2.0 :001 > Person.new('Łukasz Strzebińczyk', 26).adult?
=> true
Or is the persons name hard to pronounce? 

class Person
  def initialize(name, age)
    @name = name
    @age  = age
  end

  def adult?
    @age >= 18
  end

  def name_hard_to_pronounce?
    @name.downcase.include?('strz')
  end
end

2.2.0 :001 > Person.new('Łukasz Strzebińczyk', 26).name_hard_to_pronounce?
=> true
From here, the universe is ready to be explored!
However...
When working on rails application, we often encounter things like this: 

class User < ActiveRecord::Base
end
It looks a hell lot like 

class User
end
And it does magic.
Except it does not.
It's just a very cleverly written code, which does a lot of work for you
so you are free to work on something productive
If we have an application with posts in database schema: 

create_table "posts", force: true do |t|
  t.string   "title"
  t.text     "content"
  t.integer  "user_id"
end
Then the corresponding ActiveRecord class: 

class Post < ActiveRecord::Base
  belongs_to :user
end
does roughly this: 

class Post
  attr_accessor :title, :content, :user_id  

  def initialize(params)
    @title   = params[:title]
    @content = params[:content]
    @user_id = params[:user_id]
  end

  def user
    User.find_by(id: @user_id)
  end
end
Just remember there is no magic involved.
All the code you see is just objects and methods.
